# A012 CSV 批次匯入管理

## 功能說明
管理後台提供獨立的「CSV 匯入」TAB，集中管理四種資料的 CSV 批次匯入：員工、部門、供應商、便當菜單。每種類型皆提供 CSV 範本下載，匯入時進行格式與重複資料驗證，以 5000 筆為一批次進行 Transaction 處理，成功後將匯入結果回傳前端顯示。

## 使用者角色
- 系統管理員

## 功能需求

### 系統管理員
- 下載各類型 CSV 範本（含正確表頭，第一行為表頭）
- 上傳員工 CSV 進行批次匯入
- 上傳部門 CSV 進行批次匯入
- 上傳供應商 CSV 進行批次匯入
- 上傳便當菜單 CSV 進行批次匯入
- 匯入成功後查看已匯入的資料清單

## CSV 欄位規格

### 員工 CSV（employees）
表頭：`username,name,email,department_id`
- 匯入的員工一律設為：is_admin = false、is_active = true
- ID 由系統自動產生

### 部門 CSV（departments）
表頭：`name`
- ID 由系統自動產生

### 供應商 CSV（suppliers）
表頭：`name,email,phone,contact_person,business_registration_no`
- ID 由系統自動產生

### 便當菜單 CSV（menus）
表頭：`supplier_id,name,category,description,price,valid_from,valid_to`
- ID 由系統自動產生

## 驗證規則
1. **編碼驗證**：CSV 檔案必須為 UTF-8 編碼，非 UTF-8 則拒絕匯入並回傳錯誤
2. **表頭驗證**：第一行表頭必須與規格完全一致，不一致則拒絕匯入並回傳錯誤
3. **欄位完整性**：每列所有必填欄位不可為空
4. **重複資料檢查**：
   - 員工：username 與 email 不可與既有資料重複，亦不可與同批次內其他列重複
   - 部門：name 不可與既有資料重複，亦不可與同批次內其他列重複
   - 供應商：business_registration_no 不可與既有資料重複，亦不可與同批次內其他列重複
   - 便當菜單：同一 supplier_id + name 組合不可與既有資料重複，亦不可與同批次內其他列重複

## 批次處理規則
- 跳過第一行（表頭），從第二行起處理資料
- 以 5000 筆為一個 Transaction 批次
- 同一批次內遇到任何錯誤（驗證失敗或 DB 錯誤），整批 Rollback
- Rollback 後通知前端：目前處理至哪一行（CSV 行號），請手動刪除已成功處理的列後重新上傳
- 所有批次均成功才算完整匯入完成

## 成功後行為
- 將本次匯入成功的所有資料回傳給前端並顯示於清單

## API 路徑

| 方法 | 路徑 | 說明 |
|---|---|---|
| GET | /api/admin/import/template/{type} | 下載 CSV 範本（type: employees / departments / suppliers / menus） |
| POST | /api/admin/import/employees | 批次匯入員工 CSV |
| POST | /api/admin/import/departments | 批次匯入部門 CSV |
| POST | /api/admin/import/suppliers | 批次匯入供應商 CSV |
| POST | /api/admin/import/menus | 批次匯入便當菜單 CSV |

## 流程說明

### 下載 CSV 範本流程
1. 管理員選擇資料類型並點擊「下載範本」
2. 系統回傳對應類型的 CSV 範本檔案（第一行為表頭）
3. 管理員依範本填寫資料後儲存

### CSV 匯入流程（通用）
1. 管理員選擇資料類型，上傳 CSV 檔案
2. 系統讀取第一行，驗證表頭是否與規格一致
3. 表頭不正確則回傳錯誤，不允許匯入，流程結束
4. 表頭正確則從第二行起開始處理資料
5. 依每 5000 筆為一批次進行迴圈處理：
   - 逐列驗證欄位完整性
   - 逐列驗證資料是否與既有資料重複
   - 逐列驗證資料是否與同批次內其他列重複
   - 驗證失敗則記錄行號與錯誤原因，整批 Rollback
   - Rollback 後回傳：已處理至行號 N，請手動刪除已處理列後重新上傳
   - 流程結束
   - 驗證通過則寫入資料庫
6. 所有批次成功，回傳本次匯入成功的資料清單
