# A001 員工登入驗證與帳號管理

## 功能說明
提供公司內部員工與系統管理員透過各自的登入入口以帳號密碼進行系統登入。登入成功後依登入入口產生對應角色的 JWT Token，後端所有 API 依 JWT 的 role 欄位驗證權限。

## 使用者角色
- 員工
- 系統管理員

## 功能需求

### 員工
- 輸入帳號與密碼進行登入
- 登入失敗時查看錯誤訊息
- 登出系統
- 忘記密碼（系統產生臨時密碼並寄送至員工 Email）
- 修改自己的密碼（需驗證舊密碼，修改成功後發送 Email 通知）

### 系統管理員
- 新增員工帳號（系統產生隨機初始密碼並寄送至員工 Email）
- 以 CSV 批次匯入員工帳號（系統產生隨機初始密碼並寄送至各員工 Email）
- 修改員工資料（username、name、email、department_id、is_admin，id 不可修改）
- 停用員工帳號
- 啟用已停用的員工帳號
- 重設員工密碼（系統自動寄送 Email 通知員工）
- 修改自己的密碼（需驗證舊密碼，修改成功後發送 Email 通知）

## 登入入口

| 角色 | 登入頁面 | API |
|---|---|---|
| 一般員工 | `/login` | `POST /api/auth/login` |
| 系統管理員 | `/admin/login` | `POST /api/admin/auth/login` |

## API 路徑

| 方法 | 路徑 | 說明 |
|---|---|---|
| POST | /api/auth/login | 員工登入 |
| POST | /api/auth/logout | 員工登出 |
| POST | /api/auth/forgot-password | 忘記密碼（寄送臨時密碼至 Email） |
| PATCH | /api/auth/change-password | 修改密碼（需帶 JWT，驗證舊密碼） |
| POST | /api/auth/refresh | 員工用 Refresh Token 換新 Access Token |
| POST | /api/admin/auth/login | 管理員登入 |
| PATCH | /api/admin/auth/change-password | 管理員修改自己的密碼（需帶 JWT，驗證舊密碼） |
| POST | /api/admin/auth/refresh | 管理員用 Refresh Token 換新 Access Token |
| GET | /api/admin/employees | 查詢員工清單 |
| POST | /api/admin/employees | 新增單一員工帳號 |
| POST | /api/admin/employees/import | CSV 批次匯入員工帳號 |
| PATCH | /api/admin/employees/{id} | 修改員工資料（username、name、email、department_id、is_admin） |
| PATCH | /api/admin/employees/{id}/status | 停用或啟用員工帳號 |
| PATCH | /api/admin/employees/{id}/reset-password | 管理員重設指定員工密碼 |

## 流程說明

### 員工登入流程
1. 員工於 `/login` 頁面輸入帳號與密碼
2. 系統呼叫 `POST /api/auth/login` 驗證帳號是否存在
3. 帳號不存在則顯示錯誤訊息，流程結束
4. 帳號存在則驗證密碼是否正確
5. 密碼錯誤則顯示錯誤訊息，流程結束
6. 密碼正確則檢查帳號是否啟用中
7. 帳號已停用則顯示停用訊息，流程結束
8. 帳號正常則產生 role=employee 的 JWT Token，導向系統首頁

### 員工登出流程
1. 員工點擊登出
2. 前端呼叫 `POST /api/auth/logout`，帶入 Access Token 與 Refresh Token
3. 後端將該員工的 Refresh Token 標記為作廢（is_revoked = true）
4. 前端清除本地 Access Token 與 Refresh Token
5. 導向登入頁面

### 管理員登入流程
1. 管理員於 `/admin/login` 頁面輸入帳號與密碼
2. 系統呼叫 `POST /api/admin/auth/login` 驗證帳號是否存在
3. 帳號不存在則顯示錯誤訊息，流程結束
4. 帳號存在則驗證密碼是否正確
5. 密碼錯誤則顯示錯誤訊息，流程結束
6. 密碼正確則檢查帳號是否啟用中
7. 帳號已停用則顯示停用訊息，流程結束
8. 帳號正常則產生 role=admin 的 JWT Token，導向管理後台首頁

### 忘記密碼流程
1. 員工於登入頁面點擊「忘記密碼」
2. 輸入帳號對應的 Email
3. 系統查詢 Email 是否存在
4. Email 不存在則顯示錯誤訊息，流程結束
5. Email 存在則產生臨時密碼並更新 password_hash
6. 系統將臨時密碼寄送至該 Email
7. 員工使用臨時密碼登入後，建議至修改密碼頁面更新密碼

### 修改密碼流程
1. 員工登入後進入修改密碼頁面
2. 輸入舊密碼與新密碼
3. 系統驗證新密碼格式（8～16 位，須含大小寫字母）
4. 格式不符則顯示錯誤訊息，流程結束
5. 系統驗證舊密碼是否正確
6. 舊密碼錯誤則顯示錯誤訊息，流程結束
7. 舊密碼正確則更新 password_hash
8. 系統寄送 Email 通知使用者密碼已修改
9. 後端將該員工所有 Refresh Token 標記為作廢（is_revoked = true）
10. 前端清除本地 Access Token 與 Refresh Token 並導向登入頁面
11. 員工重新登入後，系統核發全新的 Access Token 與 Refresh Token

### 管理員修改密碼流程
1. 管理員登入後進入修改密碼頁面
2. 輸入舊密碼與新密碼
3. 系統驗證新密碼格式（8～16 位，須含大小寫字母）
4. 格式不符則顯示錯誤訊息，流程結束
5. 系統驗證舊密碼是否正確
6. 舊密碼錯誤則顯示錯誤訊息，流程結束
7. 舊密碼正確則更新 password_hash
8. 系統寄送 Email 通知管理員密碼已修改
9. 後端將該管理員所有 Refresh Token 標記為作廢（is_revoked = true）
10. 前端清除本地 Access Token 與 Refresh Token 並導向管理員登入頁面

### 查詢員工清單流程
1. 管理員進入員工管理頁面
2. 系統呼叫 `GET /api/admin/employees` 回傳所有員工清單
3. 管理員可依此清單執行停用、重設密碼等操作

### 新增單一員工帳號流程
1. 管理員輸入員工資料（username、name、email、department_id）
2. 系統驗證 username 與 email 是否已存在
3. 已存在則顯示錯誤訊息，流程結束
4. 系統建立員工帳號，產生隨機初始密碼
5. 系統將初始密碼寄送至員工 Email
6. 顯示新增成功

### CSV 批次匯入員工帳號流程
1. 管理員進入 CSV 匯入頁面，系統顯示支援的 CSV 格式說明（欄位：username、name、email、department_id）
2. 管理員上傳 CSV 檔案
3. 系統檢查 CSV 欄位是否符合格式（欄位名稱與數量）
4. 欄位不正確則顯示錯誤提示，不允許匯入
5. 欄位正確則逐列驗證資料是否完整
6. 驗證失敗的列記錄行數與錯誤原因，跳過不匯入
7. 驗證通過的列建立員工帳號，初始密碼隨機產生
8. 系統將初始密碼寄送至各員工 Email
9. 回傳匯入結果（成功筆數、失敗筆數與錯誤明細，錯誤明細含行數與錯誤原因）

### 修改員工資料流程
1. 管理員從員工清單選擇目標員工
2. 系統顯示該員工現有資料（id 顯示但不可編輯）
3. 管理員修改可編輯欄位（username、name、email、department_id、is_admin）並送出
4. 系統驗證 username 與 email 是否與其他員工重複
5. 重複則顯示錯誤訊息，不儲存
6. 不重複則更新資料庫並顯示修改成功

### 停用／啟用員工帳號流程
1. 管理員從員工清單選擇目標員工
2. 選擇停用或啟用操作
3. 系統依操作將 is_active 設為 false 或 true
4. 顯示操作成功

### 管理員重設員工密碼流程
1. 管理員選擇欲重設密碼的員工
2. 輸入新密碼
3. 系統驗證新密碼格式（8～16 位，須含大小寫字母）
4. 格式不符則顯示錯誤訊息，流程結束
5. 系統更新該員工的 password_hash
6. 系統自動寄送 Email 通知員工密碼已被重設

### Token 自動刷新流程
1. 前端發送 API 請求，附帶 Access Token
2. 後端驗證 Access Token，若過期或無效則回傳 401
3. 前端攔截 401，確認本地是否有 Refresh Token
4. 若無 Refresh Token，清除所有 Token 並導回登入頁，流程結束
5. 若有 Refresh Token，前端呼叫 Refresh API（`POST /api/auth/refresh` 或 `POST /api/admin/auth/refresh`）
6. 後端驗證 Refresh Token 是否有效
7. Refresh Token 無效或過期，後端回傳 401，前端清除所有 Token 並導回登入頁，流程結束
8. Refresh Token 有效，後端核發新的 Access Token 與新的 Refresh Token（舊 Refresh Token 作廢）
9. 前端以新的 Access Token 重試原請求
10. 請求成功，使用者無感繼續操作

## JWT 規則
- JWT payload 包含 `role` 欄位
- 員工登入後 role 固定為 `employee`
- 管理員登入後 role 固定為 `admin`
- 後端所有 API 依 JWT 的 `role` 驗證權限，role 不符者回傳 403
- 登入成功後同時核發 Access Token（短效期）與 Refresh Token（長效期）
- Access Token 過期或驗證失敗時，後端回傳 401
- 前端收到 401 時，若持有有效 Refresh Token，自動呼叫 Refresh API 換取新的 Access Token，然後重試原請求
- 若 Refresh Token 亦過期或無效，前端清除所有 Token 並導回登入頁
- Refresh Token 僅能使用一次（使用後作廢，同時核發新的 Refresh Token）

## 資料表

### employees（員工資料表）
- id（PK）
- username（帳號）
- password_hash（密碼雜湊）
- name（姓名）
- email（Email，用於寄送密碼通知）
- is_admin（是否為管理員）
- is_active（是否啟用）
- department_id（FK → departments.id，所屬部門）
- created_at（建立時間）
- updated_at（更新時間）

### refresh_tokens（Refresh Token 資料表）
- id（PK）
- employee_id（FK → employees.id）
- token_hash（Token 雜湊值）
- expires_at（過期時間）
- is_revoked（是否已作廢）
- created_at（建立時間）
